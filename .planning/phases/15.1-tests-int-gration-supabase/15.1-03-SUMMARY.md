## Decisions Made

- **Cookie format corrigé**: `@supabase/ssr` v0.6.1 `combineChunks()` lit le cookie JSON direct (ou prefixe `base64-` + base64URL). La version initiale utilisait `Buffer.from(json).toString('base64')` (base64 standard) qui n'est pas reconnu. Fix: `encodeURIComponent(cookieValue)` — confirmé fonctionnel via test direct Node.js.

- **Variables partagées entre describes**: `let createdModelId` et `let createdImageId` au niveau module permettent d'enchainer POST → GET/PUT → DELETE dans l'ordre sans répéter le setup. Vitest intégration = `singleFork` → ordre séquentiel garanti.

- **JPEG minimal 22 bytes**: Buffer codé en dur `[0xFF, 0xD8, ...]` évite toute dépendance externe pour tester l'upload Storage. Supabase local accepte le fichier (pas de validation de contenu côté bucket).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Correction format cookie @supabase/ssr dans auth.ts**

- **Found during:** Task 1 (premier run de test — tous les tests retournaient 401)
- **Issue:** `auth.ts` encodait le cookie JSON en base64 standard (`Buffer.from(json).toString('base64')`). `@supabase/ssr@0.6.1` `combineChunks()` ne reconnaît que la valeur JSON brute ou le préfixe `base64-` (base64URL), pas base64 standard — résultat: `getUser()` → "Auth session missing!" → 401 sur toutes les routes admin.
- **Fix:** Remplacé par `encodeURIComponent(cookieValue)` (JSON direct URL-encodé). Next.js décode l'URI quand il lit les cookies, `@supabase/ssr` reçoit le JSON brut.
- **Files modified:** `src/__tests__/integration/helpers/auth.ts`
- **Vérification:** `server.auth.getUser()` retourne `admin@test.mobelunique.fr`, tous les 13 tests admin-models passent.
- **Committed in:** `96457c6` (Task 1 commit)
