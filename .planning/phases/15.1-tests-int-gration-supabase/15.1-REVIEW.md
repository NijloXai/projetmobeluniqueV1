## Resume

Phase 15.1 ajoute une infrastructure de tests d'integration complete pour l'application Next.js + Supabase : helpers (auth, seed, cleanup, client admin), 7 fichiers de test couvrant toutes les routes API (admin models, fabrics, generate, visuals, auth/RLS, public models, simulate), plus la configuration Supabase locale (config.toml, migration SQL, seed SQL) et la config vitest multi-projets.

L'architecture de test est globalement solide : isolation via `singleFork`, seed idempotent avec UUIDs fixes, cleanup cible par slug/id dans les `afterAll`, et bonne couverture des cas nominaux et d'erreur (401, 400, 404, 403). Cependant, une faille de securite dans les RLS policies et plusieurs problemes de robustesse ont ete identifies.

## Critical Issues

### CR-01 : RLS policies trop permissives -- les modeles/tissus inactifs sont lisibles par anon

**File :** `supabase/migrations/20260410000000_schema.sql:56-59`
**Issue :** Les policies `models_public_read`, `fabrics_public_read`, etc. utilisent `USING (true)` ce qui permet a n'importe quel client non authentifie de lire TOUTES les lignes, y compris les modeles inactifs (`is_active = false`) et les tissus inactifs. Le filtrage `is_active = true` est fait uniquement au niveau applicatif (dans les routes API), mais un client Supabase direct (ou une requete via PostgREST) peut contourner ce filtre et acceder aux donnees non publiees. Cela contredit l'intention documentee dans CLAUDE.md : "Roma Inactif ne doit PAS etre visible sur les routes publiques."

De plus, le test `auth-rls.integration.test.ts` ligne 96-107 pretend tester "les routes publiques ne montrent pas les modeles inactifs" mais utilise `adminClient` (service_role) avec un filtre `.eq('is_active', true)` -- cela ne teste pas du tout les RLS policies, juste un filtre SQL.

**Fix :**
```sql
-- Remplacer les policies public_read par des policies filtrees
CREATE POLICY "models_public_read" ON models
  FOR SELECT USING (is_active = true);

CREATE POLICY "fabrics_public_read" ON fabrics
  FOR SELECT USING (is_active = true);

-- model_images et generated_visuals peuvent rester USING (true)
-- car ils sont lies par FK a des modeles/tissus deja filtres,
-- ou ajouter un filtre via subquery :
CREATE POLICY "generated_visuals_public_read" ON generated_visuals
  FOR SELECT USING (is_published = true);
```

Et corriger le test RLS pour utiliser `anonClient` au lieu de `adminClient` :
```typescript
import { anonClient } from '../helpers/supabase-admin'

it('les routes publiques ne montrent pas les modeles inactifs', async () => {
  const { data } = await anonClient
    .from('models')
    .select('name, is_active')
  const names = (data ?? []).map((m: { name: string }) => m.name)
  expect(names).not.toContain('Roma Inactif')
})
```

## Warnings

### WR-01 : Incoherence de valeurs par defaut entre seed.ts et auth.ts

**File :** `src/__tests__/integration/helpers/auth.ts:5-6`
**Issue :** `seed.ts` definit des fallbacks pour `TEST_ADMIN_EMAIL` et `TEST_ADMIN_PASSWORD` (lignes 3-4 : `?? 'admin@test.mobelunique.fr'` et `?? 'test-admin-secure-2024!'`), mais `auth.ts` utilise des non-null assertions (`process.env.TEST_ADMIN_EMAIL!`) sans fallback. Si ces variables ne sont pas dans `.env.test.local`, `seedAdminUser()` creera l'utilisateur avec les valeurs par defaut, mais `loginAsAdmin()` passera `undefined` a `signInWithPassword`, causant une erreur cryptique au lieu d'un message clair.

**Fix :**
```typescript
// auth.ts — utiliser les memes fallbacks que seed.ts
const TEST_ADMIN_EMAIL = process.env.TEST_ADMIN_EMAIL ?? 'admin@test.mobelunique.fr'
const TEST_ADMIN_PASSWORD = process.env.TEST_ADMIN_PASSWORD ?? 'test-admin-secure-2024!'

export async function loginAsAdmin(): Promise<{ accessToken: string; cookieHeader: string }> {
  const { data, error } = await anonClient.auth.signInWithPassword({
    email: TEST_ADMIN_EMAIL,
    password: TEST_ADMIN_PASSWORD,
  })
  // ...
}
```

### WR-02 : Dependance entre tests via variables mutables partagees

**File :** `src/__tests__/integration/routes/admin-fabrics.integration.test.ts:11`
**File :** `src/__tests__/integration/routes/admin-models.integration.test.ts:14-15`
**Issue :** Les variables `createdFabricId` et `createdModelId`/`createdImageId` sont initialisees dans un test `POST` et utilisees dans les tests suivants (`GET [id]`, `PUT [id]`, `DELETE [id]`). Si le test POST echoue, toutes les assertions suivantes echouent avec des erreurs confuses (`params: { id: undefined }`). Cela rend le diagnostic des echecs difficile. Ce pattern est courant dans les tests d'integration sequentiels mais merite une guard clause.

**Fix :** Ajouter un guard au debut de chaque `describe` qui depend de la variable :
```typescript
describe('GET /api/admin/fabrics/[id]', () => {
  it('retourne le tissu par id', async () => {
    expect(createdFabricId).toBeDefined() // Guard: echoue explicitement si POST a rate
    await testApiHandler({
      // ...
    })
  })
})
```

Ou utiliser `it.skipIf(!createdFabricId)` pour marquer clairement les tests sautes.

### WR-03 : Duplication seed SQL / seed TypeScript

**File :** `supabase/seed.sql:1-21`
**File :** `src/__tests__/integration/helpers/seed.ts:28-59`
**Issue :** Les memes donnees de test (memes UUIDs, memes valeurs) sont definies a la fois dans `seed.sql` (execute par `supabase db reset`) et dans `seedTestData()` (execute dans `beforeAll`). Cela cree un risque de desynchronisation si l'un est modifie sans l'autre. Actuellement `seedTestData()` utilise `upsert` donc pas de conflit, mais la duplication est une source de bugs futurs. De plus, si `supabase db reset` n'est pas execute avant les tests (par ex. CI sans Docker), le `seed.sql` n'est jamais utilise et les tests dependent entierement du seed TS.

**Fix :** Choisir une seule source de verite. L'option recommandee est de garder `seed.sql` minimal (juste les buckets Storage qui ne peuvent pas etre crees via l'API JS) et de deplacer toutes les donnees de test dans `seedTestData()` TS. Alternativement, supprimer le contenu duplique de `seed.sql` et ne garder que les `INSERT INTO storage.buckets` (deja dans la migration).

## Info

### IN-01 : Fichier cleanup.ts jamais importe (code mort)

**File :** `src/__tests__/integration/helpers/cleanup.ts:1-37`
**Issue :** Les fonctions `cleanupTestData()` et `cleanupAll()` sont exportees mais jamais importees dans aucun fichier de test. Chaque suite de test fait son propre cleanup dans `afterAll`. Ce fichier constitue du code mort.

**Fix :** Soit supprimer le fichier, soit l'importer dans les `afterAll` des suites de test pour centraliser la logique de nettoyage (recommande pour la maintenabilite).

### IN-02 : project_id dans config.toml semble auto-genere

**File :** `supabase/config.toml:5`
**Issue :** `project_id = "agent-ab074def"` ressemble a un identifiant auto-genere a partir du nom de worktree (`agent-a6a71aa0`). Ce nom n'est pas explicite pour le projet Mobel Unique et pourrait preter a confusion.

**Fix :** Renommer en quelque chose de descriptif :
```toml
project_id = "mobel-unique-local"
```

### IN-03 : Donnees de seed duplicees dans seed.sql (deja dans la migration)

**File :** `supabase/seed.sql:5-21`
**Issue :** Les `INSERT INTO storage.buckets` sont deja dans la migration (`20260410000000_schema.sql:74-77`). Le `seed.sql` ne contient que les donnees de test (models, fabrics, etc.) qui sont egalement presentes dans `seedTestData()` TS. Si le `seed.sql` est destine uniquement au `supabase db reset`, les donnees de test y sont redondantes avec le seed TS.

**Fix :** Voir WR-03 — garder une seule source de verite.
