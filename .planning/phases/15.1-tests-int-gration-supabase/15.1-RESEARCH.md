## Summary

Cette phase met en place une couche de tests d'intégration Vitest contre une instance Supabase locale (Docker via Supabase CLI). Les 183 tests unitaires existants (Phase 15) mockent entièrement Supabase — cette phase ajoute une couche orthogonale qui teste les vrais comportements : auth JWT, RLS, Storage, et les 20 routes API.

Le pattern central consiste à : (1) démarrer Supabase CLI localement (`supabase start`), (2) exécuter les migrations + un seed SQL, (3) utiliser `next-test-api-route-handler` (NTARH) pour invoquer les handlers de routes dans un contexte Next.js réel, (4) utiliser deux clients Supabase (service_role pour setup/teardown, anon+JWT pour tests auth).

La contrainte principale est l'absence de Docker sur la machine actuelle — le plan doit documenter les prérequis et fournir un script `test:integration` qui gère le lifecycle Supabase. Les tests d'intégration doivent s'exécuter séquentiellement (`fileParallelism: false`) pour éviter les conflits de FK Supabase.

**Recommandation principale:** Utiliser `next-test-api-route-handler` v5.0.4 + Vitest projects (séparation unit/integration) + Supabase service_role client pour seed/teardown + `signInWithPassword` pour obtenir les vrais JWT admin.

      routes/

### Pattern 1: Vitest Projects — Séparation Unit / Integration

**What:** Un seul `vitest.config.ts` avec deux projects — `unit` (tests existants) et `integration` (nouveaux tests).

**When to use:** Toujours — évite que les tests d'intégration lents bloquent le feedback rapide des tests unitaires.

**Example:**
```typescript
// vitest.config.ts — Source: https://vitest.dev/guide/projects (VERIFIED)
import { defineConfig } from 'vitest/config'
import react from '@vitejs/plugin-react'
import path from 'path'

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: { '@': path.resolve(__dirname, './src') },
  },
  test: {
    projects: [
      {
        extends: true,
        test: {
          name: 'unit',
          environment: 'happy-dom',
          setupFiles: ['./src/__tests__/setup.ts'],
          include: ['src/**/*.test.{ts,tsx}'],
          exclude: ['src/**/*.integration.test.ts'],
        },
      },
      {
        extends: true,
        test: {
          name: 'integration',
          environment: 'node',
          setupFiles: ['./src/__tests__/integration/setup.ts'],
          include: ['src/**/*.integration.test.ts'],
          fileParallelism: false,   // CRITIQUE : FK contraintes Supabase
          testTimeout: 30000,
        },
      },
    ],
  },
})
```

### Pattern 2: Two-Client Strategy (service_role + anon)

**What:** Deux clients Supabase dans les tests — un service_role pour seed/teardown (bypass RLS), un anon pour tester les comportements réels des routes.

**When to use:** Pour tous les tests d'intégration qui interagissent avec Supabase.

**Example:**
```typescript
// src/__tests__/integration/helpers/supabase-admin.ts
import { createClient } from '@supabase/supabase-js'
import type { Database } from '@/types/database'

// Service role — bypass RLS, pour seed + teardown uniquement
export const adminClient = createClient<Database>(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,
  {
    auth: {
      autoRefreshToken: false,
      persistSession: false,  // CRITIQUE : évite partage session avec client anon
    },
  }
)

// Client anon — simule l'app réelle
export const anonClient = createClient<Database>(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
  {
    auth: { persistSession: false },
  }
)
```

### Pattern 3: Auth Flow — Obtenir un JWT réel pour les routes admin

**What:** `signInWithPassword` avec le user créé par seed → extract `access_token` → passer en Authorization header ou Cookie pour les routes admin.

**When to use:** Tous les tests des routes admin (D-07, D-08).

**Example:**
```typescript
// src/__tests__/integration/helpers/auth.ts
import { anonClient } from './supabase-admin'

export async function loginAsAdmin(): Promise<{ accessToken: string; cookieHeader: string }> {
  const { data, error } = await anonClient.auth.signInWithPassword({
    email: process.env.TEST_ADMIN_EMAIL!,   // défini dans seed + .env.test.local
    password: process.env.TEST_ADMIN_PASSWORD!,
  })
  if (error || !data.session) throw new Error(`Auth failed: ${error?.message}`)

  const accessToken = data.session.access_token
  // Format cookie attendu par @supabase/ssr
  const cookieHeader = `sb-access-token=${accessToken}`

  return { accessToken, cookieHeader }
}
```

### Pattern 4: NTARH (next-test-api-route-handler) pour invoquer les routes

**What:** `testApiHandler` wraps le vrai handler de route et patche les internals Next.js (cookies, headers, fetch).

**When to use:** Tous les tests de routes API. NTARH doit être le PREMIER import.

**Example:**
```typescript
// src/__tests__/integration/routes/admin-models.integration.test.ts
import { testApiHandler } from 'next-test-api-route-handler' // PREMIER import OBLIGATOIRE
import * as modelsRoute from '@/app/api/admin/models/route'
import { loginAsAdmin } from '../helpers/auth'
import { adminClient } from '../helpers/supabase-admin'
import { describe, it, expect, beforeAll, afterAll } from 'vitest'

describe('POST /api/admin/models', () => {
  let authCookie: string

  beforeAll(async () => {
    const { cookieHeader } = await loginAsAdmin()
    authCookie = cookieHeader
  })

  afterAll(async () => {
    // Cleanup : truncate les données créées par ce test
    await adminClient.from('models').delete().like('slug', 'test-%')
  })

  it('crée un modèle (cas nominal)', async () => {
    await testApiHandler({
      appHandler: modelsRoute,
      async test({ fetch }) {
        const res = await fetch({
          method: 'POST',
          headers: {
            'Content-Type': 'application/json',
            Cookie: authCookie,
          },
          body: JSON.stringify({
            name: 'Test Milano',
            price: 1200,
          }),
        })
        expect(res.status).toBe(201)
        const data = await res.json()
        expect(data.slug).toBe('test-milano')
      },
    })
  })

  it('retourne 401 sans token', async () => {
    await testApiHandler({
      appHandler: modelsRoute,
      async test({ fetch }) {
        const res = await fetch({ method: 'GET' })
        expect(res.status).toBe(401)
      },
    })
  })
})
```

### Pattern 5: Seed SQL + supabase db reset

**What:** `supabase/seed.sql` peuplé avec un user admin et des données de base. Exécuté automatiquement à `supabase start` et `supabase db reset`.

**When to use:** Initialisation de l'environnement de test.

**Example:**
```sql
-- supabase/seed.sql
-- User admin pour les tests (hash bcrypt du mot de passe)
INSERT INTO auth.users (id, email, encrypted_password, email_confirmed_at, role, aud, created_at, updated_at)
VALUES (
  'a0000000-0000-0000-0000-000000000001',
  'admin@test.mobelunique.fr',
  crypt('test-password-secure', gen_salt('bf')),
  now(),
  'authenticated',
  'authenticated',
  now(),
  now()
);

-- Données de test : 1 modèle actif, 1 tissu actif
INSERT INTO models (id, name, slug, price, is_active, created_at)
VALUES ('b0000000-0000-0000-0000-000000000001', 'Milano Test', 'milano-test', 1200, true, now());

INSERT INTO fabrics (id, name, slug, category, is_active, created_at)
VALUES ('c0000000-0000-0000-0000-000000000001', 'Velours Test', 'velours-test', 'velours', true, now());
```

**IMPORTANT:** La création d'un user admin dans auth.users via SQL raw nécessite le schéma `auth` — qui est géré par Supabase Auth Service, pas accessible via le client `@supabase/supabase-js` en mode anon. Utiliser `adminClient.auth.admin.createUser()` dans le globalSetup ou dans le seed via SQL avec le bon hash.

**Recommandation:** Créer le user admin via `adminClient.auth.admin.createUser()` dans le globalSetup (ou dans un `beforeAll` du premier test) — plus portable que le SQL raw qui dépend de la structure interne de auth.users.

### Pattern 6: Cleanup entre suites de tests

**What:** `afterAll` par fichier de test, utilisant le service_role client pour truncate les données créées par le fichier.

**When to use:** Fin de chaque fichier de test d'intégration.

**Example:**
```typescript
afterAll(async () => {
  // Supprimer données créées — le service_role bypasse RLS et FK order
  await adminClient.from('generated_visuals').delete().like('id', '%') // si created in test
  await adminClient.from('models').delete().like('slug', 'test-%')
  await adminClient.from('fabrics').delete().like('slug', 'test-%')
})
```

### Anti-Patterns to Avoid

- **Importer le handler APRÈS testApiHandler:** NTARH doit être le premier import, sinon Next.js n'est pas patché correctement — les appels à `cookies()` de `next/headers` échouent.
- **Tests parallèles avec Supabase:** `fileParallelism: false` est obligatoire — les contraintes FK de Supabase causent des erreurs non-déterministes en parallèle.
- **Utiliser happy-dom pour les tests d'intégration:** Les tests d'intégration testent des handlers Node.js — l'environnement doit être `node`, pas `happy-dom`.
- **Service_role avec persistSession true:** Le service_role client partage la session auth avec l'anon client si `persistSession` n'est pas `false` — les tests RLS deviennent non-fiables.
- **Supprimer auth.users via SQL direct:** Utiliser `adminClient.auth.admin.deleteUser(id)` — SQL direct sur `auth.users` peut violer les contraintes internes de GoTrue.
- **supabase db reset dans les tests:** Trop lent (20-30s), réservé au setup initial. Utiliser truncate ciblé dans `afterAll`.

## Don't Hand-Roll

| Problème | Ne pas construire | Utiliser à la place | Pourquoi |
|---------|-------------|-------------|-----|
| Tester App Router route handlers | Wrapper maison avec NextRequest + server mock | `next-test-api-route-handler` | Les internals Next.js (headers(), cookies(), params) ne peuvent pas être simulés manuellement — NTARH utilise les vrais résolveurs Next.js |
| Créer un user admin pour les tests | SQL raw dans auth.users | `adminClient.auth.admin.createUser()` | GoTrue gère le hashing, la confirmation email, les sessions — bypasser via SQL cause des états incohérents |
| Configurer l'instance locale Supabase | Docker Compose custom | Supabase CLI (`supabase init` + `supabase start`) | CLI gère le versioning Postgres, Auth, Storage, Studio, migrations — impossible à reproduire manuellement sans bugs |
| Cleanup entre tests | DROP TABLE / recreate schema | Truncate ciblé via service_role client | Reset migrations prend 20-30s ; truncate prend < 100ms |

## Common Pitfalls

### Pitfall 1: NTARH pas en premier import

**What goes wrong:** `Error: cookies() was called outside a request scope` ou `headers() is not defined` lors des tests des routes qui appellent `createClient()` depuis `@/lib/supabase/server`.

**Why it happens:** Next.js patche les globals (`fetch`, `headers`, `cookies`) au démarrage. Si NTARH n'est pas importé en premier, ces patches ne sont pas appliqués.

**How to avoid:** Toujours mettre `import { testApiHandler } from 'next-test-api-route-handler'` comme première ligne de chaque fichier de test d'intégration.

**Warning signs:** Erreur `cookies() was called outside a request scope` ou `Dynamic server usage` lors de l'exécution des tests.

### Pitfall 2: Tests d'intégration en parallèle

**What goes wrong:** Des erreurs aléatoires `23503 foreign key violation` ou des données manquantes dans certains tests.

**Why it happens:** Vitest parallélise les fichiers de test par défaut. Si deux fichiers créent puis suppriment les mêmes entités de seed simultanément, les FK cascades interfèrent.

**How to avoid:** `fileParallelism: false` dans le project `integration` de vitest.config.ts.

**Warning signs:** Tests qui passent seuls mais échouent en suite complète.

### Pitfall 3: Session partagée entre service_role et anon client

**What goes wrong:** Le service_role client "voit" les auth events du client anon — les politiques RLS ne sont pas testées correctement (le client devrait être non-authentifié mais se retrouve authentifié).

**Why it happens:** `@supabase/supabase-js` partage l'état auth via localStorage/sessionStorage. En environnement Node (Vitest), il utilise un store global partagé.

**How to avoid:** `auth: { persistSession: false, autoRefreshToken: false }` sur le service_role client. Créer un nouvel `anonClient` par test si nécessaire.

**Warning signs:** Les tests RLS "access refusé" passent alors qu'ils devraient échouer.

### Pitfall 4: Clés locales Supabase hardcodées

**What goes wrong:** Les clés locales Supabase (anon key, service_role key) sont toujours les mêmes pour tous les projets CLI (`supabase start` génère des JWT signés avec le secret par défaut `super-secret-jwt-token-with-at-least-32-characters-long`). Si elles sont commitées, elles ne donnent pas accès à la prod — mais c'est une mauvaise pratique.

**How to avoid:** Stocker dans `.env.test.local` (gitignored) et documenter les valeurs dans le README.

**Warning signs:** `.env.test.local` absent dans `.gitignore`.

### Pitfall 5: Buckets Storage non créés dans l'instance locale

**What goes wrong:** Les tests Storage (`upload`, `remove`, `getPublicUrl`) échouent avec `Bucket not found`.

**Why it happens:** `supabase start` crée l'instance vierge — les buckets doivent être créés via les migrations ou le seed (la migration SQL qui les crée n'existe peut-être pas).

**How to avoid:** Ajouter une migration SQL pour créer les 4 buckets (`model-photos`, `fabric-swatches`, `fabric-references`, `generated-visuals`) avec leurs politiques, ou les créer via `adminClient.storage.createBucket()` dans le globalSetup.

**Warning signs:** Erreur `StorageError: Bucket not found` dans les tests Storage.

### Pitfall 6: Route /api/simulate utilise rate-limiting en mémoire

**What goes wrong:** Après 5 appels dans les tests, simulate retourne 429.

**Why it happens:** La route simulate a un rate-limiter en mémoire (Map) sur l'IP — les tests répétés depuis la même IP déclenchent le limit.

**How to avoid:** Les tests simulate ne doivent pas dépasser 5 appels avec la même IP simulée, ou patcher l'IP dans les tests (header `x-forwarded-for` différent par test).

**Warning signs:** Status 429 dans les tests simulate après le 5ème appel.

### Pitfall 7: Route IA (generate, generate-all) appelle le vrai provider

**What goes wrong:** Si `NANO_BANANA_API_KEY` est définie dans l'env de test, les tests appellent Gemini — lents, coûteux, non-déterministes.

**Why it happens:** Le factory pattern `getIAService()` vérifie `NANO_BANANA_API_KEY` — si présente, utilise Nano Banana.

**How to avoid:** S'assurer que `NANO_BANANA_API_KEY` est absente (ou vide) dans `.env.test.local`. Le Mock Sharp sera utilisé automatiquement.

**Warning signs:** Tests de génération qui prennent > 5s.

import { adminClient } from './supabase-admin'

export async function seedAdminUser() {
  // Vérifie si déjà créé (idempotent)
  const { data: existing } = await adminClient.auth.admin.listUsers()
  const exists = existing?.users?.find(u => u.email === 'admin@test.mobelunique.fr')
  if (exists) return exists

  const { data, error } = await adminClient.auth.admin.createUser({

    email_confirm: true,  // Confirme immédiatement — pas de vérification email en local
  })
  if (error) throw new Error(`Seed user failed: ${error.message}`)
  return data.user
}

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | oui | `signInWithPassword` réel — tester JWT valide vs expiré vs absent |
| V3 Session Management | oui | Cookie HttpOnly — vérifier que les routes admin rejettent les sessions invalides |
| V4 Access Control | oui | RLS Supabase — vérifier politiques anon vs authenticated vs service_role |
| V5 Input Validation | oui | Zod schemas — tester payloads invalides sur toutes les routes POST/PUT |
| V6 Cryptography | non | Supabase gère le chiffrement — pas de crypto custom dans les routes |

### Known Threat Patterns for This Stack

| Pattern | STRIDE | Standard Mitigation dans les tests |
|---------|--------|---------------------|
| Route admin accessible sans token | Elevation of Privilege | Test 401 sans Cookie/Authorization header |
| RLS bypass via SQL injection dans les params | Tampering | Tester avec des UUIDs malformés, caractères spéciaux dans slug |
| Accès aux données d'un autre user | Information Disclosure | Vérifier RLS : user anon ne voit pas les modèles is_active=false |
| Upload fichier non autorisé dans bucket privé (fabric-references) | Tampering | Tester upload sans auth → Storage policy doit rejeter |

## Open Questions

1. **Migrations SQL existantes**
   - What we know: Le projet a 4 tables et 4 buckets en production Supabase. Il n'y a pas de dossier `supabase/` — les migrations n'ont pas été versionnées.
   - What's unclear: Est-ce que les migrations SQL (CREATE TABLE, RLS policies, Storage policies) sont documentées quelque part, ou faut-il les recréer depuis `supabase/schema.sql` exporté du projet Supabase distant ?
   - Recommendation: Wave 0 doit inclure `supabase db pull` depuis le projet distant pour capturer les migrations actuelles.

2. **Env vars dans Vitest — quelle méthode utiliser pour `.env.test.local`**
   - What we know: Vitest hérite de Vite et peut utiliser `loadEnv`. Node 22+ a `loadEnvFile`. `dotenv` fonctionne aussi.
   - What's unclear: Quelle méthode est la plus propre avec vitest 3.2.4 et le projet sans mode Vite.
   - Recommendation: Utiliser `dotenv` dans `src/__tests__/integration/setup.ts` — pattern le plus explicite et compatible avec le setup file pattern existant.

3. **Middleware `updateSession` (proxy.ts) — comment tester en intégration**
   - What we know: `updateSession` est dans `src/lib/supabase/middleware.ts`, appelé depuis `proxy.ts` (nom non-standard). Ce n'est pas une route API — c'est un middleware Next.js.
   - What's unclear: NTARH ne teste pas les middlewares Next.js, seulement les route handlers. Comment tester le redirect vers /admin/login ?
   - Recommendation: Tester `updateSession` directement comme une fonction (appel direct avec un `NextRequest` mocké) ou via un test E2E Playwright (Phase 16). D-09 peut être partiellement couvert par un test unitaire du middleware lui-même.
