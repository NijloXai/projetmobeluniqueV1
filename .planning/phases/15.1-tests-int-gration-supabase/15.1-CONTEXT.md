# Phase 15.1: Tests Intégration Supabase - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Mettre en place des tests d'intégration Vitest contre une instance Supabase locale (CLI Docker) pour vérifier le comportement réel des 20 routes API, de l'auth, du Storage et des politiques RLS. Les tests unitaires avec mocks existent déjà (183 tests, Phase 15) — cette phase ajoute la couche intégration qui teste les vraies interactions Supabase.

</domain>

<decisions>
## Implementation Decisions

### Environnement de test
- **D-01:** Supabase CLI local (supabase init + supabase start) — instance Docker éphémère avec Postgres, Auth, Storage
- **D-02:** Seed SQL pour créer les données de test (user admin, modèles, tissus, images, visuels)
- **D-03:** Les tests Vitest se connectent à l'instance locale via les env vars locales (SUPABASE_URL=http://localhost:54321, clés locales)
- **D-04:** Docker requis comme prérequis — documenté dans le README si nécessaire

### Routes à couvrir
- **D-05:** Couverture exhaustive des 20 routes API qui utilisent Supabase :
  - Routes publiques (3) : GET /api/models, GET /api/models/[slug], GET /api/models/[slug]/visuals
  - Routes admin modèles (5) : GET/POST /admin/models, GET/PUT/DELETE /admin/models/[id], images, visuals
  - Routes admin tissus (3) : GET/POST /admin/fabrics, GET/PUT/DELETE /admin/fabrics/[id], categories
  - Routes admin visuels (4) : validate, publish, bulk-validate, bulk-publish, export ZIP
  - Routes IA (3) : generate, generate-all, simulate
  - Auth middleware (1) : updateSession dans proxy.ts
- **D-06:** Chaque route testée avec cas nominal + cas d'erreur (not found, validation, auth)

### Stratégie auth
- **D-07:** Vrai flow auth — créer un user admin via seed, login réel via signInWithPassword, utiliser le JWT token pour les routes admin
- **D-08:** Tester requireAdmin() avec un vrai token (pas mocké) — vérifier 401 sans token et 200 avec token valide
- **D-09:** Tester le middleware updateSession (redirect vers /admin/login si pas authentifié)

### Storage & RLS
- **D-10:** Tester les uploads dans les 4 buckets (model-photos, fabric-swatches, fabric-references, generated-visuals)
- **D-11:** Tester les suppressions Storage (cascade avant DELETE BDD)
- **D-12:** Vérifier les politiques RLS — un user non authentifié ne peut pas accéder aux données protégées, les routes publiques retournent uniquement les modèles/tissus actifs

### Claude's Discretion
- Organisation des fichiers test d'intégration (dossier séparé `src/__tests__/integration/` ou suffixe `.integration.test.ts`)
- Stratégie de cleanup entre tests (truncate tables, transactions rollback, ou re-seed)
- Ordre d'exécution des tests et gestion des dépendances inter-tests
- Configuration Vitest pour séparer tests unitaires et intégration (projects ou workspace)
- Script npm pour démarrer/arrêter Supabase local et lancer les tests d'intégration

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `vitest.config.ts` : setup complet avec happy-dom, path alias — à étendre avec un projet séparé pour l'intégration
- Pattern mock Supabase dans les tests existants — sert de référence pour comprendre les queries attendues
- `src/lib/supabase/server.ts` : createClient() avec cookies — sera remplacé par un client direct pour les tests d'intégration
- `src/types/database.ts` : types Database pour typage fort des queries

### Established Patterns
- Chaining Supabase : `.from().select().eq().single()` — le pattern standard dans toutes les routes
- Storage : `.storage.from(bucket).upload()` / `.remove()` / `.getPublicUrl()`
- Auth : `supabase.auth.getUser()` dans requireAdmin et middleware
- RLS : les routes publiques filtrent `is_active = true`, les routes admin voient tout
- Cascade DELETE : suppression fichiers Storage AVANT cascade DELETE en BDD

### Integration Points
- Les 20 routes API importent `createClient` depuis `@/lib/supabase/server` ou `requireAdmin` depuis `@/lib/supabase/admin`
- Le middleware `updateSession` dans `proxy.ts` utilise `@/lib/supabase/middleware`
- 4 buckets Storage : model-photos, fabric-swatches, fabric-references, generated-visuals
- 4 tables avec FK cascadées : models → model_images, models/fabrics → generated_visuals
