## Récapitulatif des Gaps

### Gap 1 (Bloquant partiel) : Route DELETE /api/admin/models/[id]/visuals/[visualId] non testée

La route est importée ligne 7 de `admin-models.integration.test.ts` mais jamais passée à `testApiHandler`. Sur 20 routes API du projet, 19 sont réellement testées. Cette route effectue une suppression de visuel avec nettoyage Storage dans `generated-visuals`.

**Cause racine :** Import ajouté lors de la planification (Plan 03 liste 6 imports route admin-models) mais le test correspondant n'a pas été écrit.

**Correction :** Ajouter dans `admin-models.integration.test.ts` un describe `DELETE /api/admin/models/[id]/visuals/[visualId]` avec un test qui crée un visuel via adminClient, appelle DELETE via testApiHandler, et vérifie status 200 + absence en BDD.

### Gap 2 (Partiel) : Uploads fabric-swatches et fabric-references non testés (D-10)

La route `POST /api/admin/fabrics` supporte un FormData avec un fichier swatch uploadé vers `fabric-swatches`. Les tests utilisent exclusivement `JSON.stringify()` — le chemin Storage de `fabric-swatches` n'est jamais exercé en intégration. `fabric-references` (bucket privé) est encore moins couvert.

**Impact :** D-10 ("tester les uploads dans les 4 buckets") n'est satisfait qu'à 50% (2/4 buckets). Les Success Criteria ROADMAP SC-5 est partiel.

**Correction :** Ajouter un test FormData dans `admin-fabrics.integration.test.ts` similaire au test upload image (File JPEG minimal 22 bytes) pour tester l'upload vers `fabric-swatches`.

---

### Relation entre les gaps

Ces deux gaps partagent une cause commune : les tests CRUD admin (Plan 03) se sont concentrés sur les cas nominaux JSON et ont omis les branches FormData/Storage des routes fabrics + la route de suppression visuel par ID. Ils peuvent être résolus dans un seul micro-plan correctif.
